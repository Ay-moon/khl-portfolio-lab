Senior Data Architect · Intro

Ce projet simule ce qu'une vraie banque ou société de gestion doit construire pour piloter un portefeuille d'investissement. Le problème métier est simple : comment savoir en temps réel combien tu as gagné, quel risque tu prends, et quelle décision prendre demain — le tout de façon traçable et auditable ? KHL Portfolio Lab répond à cette question, couche par couche.

Senior · Pourquoi ce schéma en étoile ?

On utilise un schéma en étoile (Dimensional Modeling) parce que c'est le standard de l'industrie BI depuis Kimball dans les années 90. Les dimensions (DimDate, DimSecurity) sont stables et peu volumineuses — tu les charges une fois. Les faits (FactTrades, FactPrice) grossissent chaque jour. Power BI fait des jointures sur les dimensions en mémoire pour les filtres et les slicers — c'est ultra-rapide. Si tu vois un schema différent en entretien, demande toujours : "Pourquoi vous n'utilisez pas le star schema ?" — ça impressionne.

SQL · Modèle de Données · Gold Layer

Le Modèle de Données Gold

Toutes les données consolidées dans un schéma SQL structuré, optimisé pour la performance analytique et la traçabilité réglementaire.

👨‍💼
Senior · C'est quoi le "Gold Layer" ?

Dans l'architecture Medallion (Bronze → Silver → Gold), le Gold Layer c'est la donnée propre, validée, enrichie — prête à être consommée par les dashboards et les applications métier. On ne touche pas au Gold depuis le code applicatif en écriture directe — uniquement via les scripts contrôlés. Pense au Gold comme à la source de vérité unique qui alimente tout le reste.

👨‍💼
Senior · Les vues SQL, pourquoi ?

Les vues SQL comme vw_PortfolioDashboardDaily ou vw_AI_LatestRecommendations sont la couche de présentation du SQL. Elles joingnent plusieurs tables et exposent exactement ce dont Power BI a besoin — sans dupliquer les données. Règle d'or : ne jamais laisser Power BI faire les jointures lui-même sur des millions de lignes. Fais-les dans SQL via des vues, et Power BI se contente de lire. Performance x10.

Senior · Fees & Slippage, expliqués simplement

Les fees (5 bps = 0.05%) c'est ce que ton courtier prend sur chaque trade. Le slippage (3 bps), c'est la différence entre le prix demandé et le prix réel d'exécution — sur les marchés liquides, le carnet d'ordres bouge pendant que ton ordre arrive. Si tu ne modélises pas ça, tes backtests sont irréalistes et optimistes. C'est l'erreur classique des débutants en quantitative finance.

# trading-sim/engine/simulator.py def execute_order(order: MarketOrder, price: float, state: PositionState) -> ExecutedTrade: # Calcul du coût réel avec frais gross = order.quantity * price fees = gross * 0.0005 # 5 bps slip = gross * 0.0003 # 3 bps net = gross + fees + slip # Mise à jour de la position if order.side == "BUY": state.update_avg_cost(price, qty) elif order.side == "SELL": if qty > state.quantity: raise ValueError("Insufficient position") return ExecutedTrade( net_amount=net, fees=fees, slippage=slip )

Analytics · Risk Management · KPI Finance

Analytics & Métriques de Risque

Le moteur de calcul Python qui transforme les transactions brutes en indicateurs de performance et de risque exploitables par les gérants.

👨‍💼
Senior · La vraie valeur ajoutée de ce module

N'importe qui peut calculer un P&L. La vraie question c'est : "est-ce que je gagne assez d'argent par rapport au risque que je prends ?" C'est ce que le Sharpe ratio mesure. Un fonds avec 10% de rendement et 5% de volatilité est bien meilleur qu'un fonds avec 15% de rendement et 20% de volatilité. Les régulateurs (AMF, ESMA) exigent ces métriques pour les fonds en distribution publique.

💰

NAV (Net Asset Value)

La valeur totale du portefeuille = Cash disponible + Valeur de marché de toutes les positions.

NAV = Cash + Σ(Qty_i × Price_i)
📈

P&L Journalier & Cumulé

Le gain ou la perte du jour (DailyPnL) et depuis le début (CumPnL). Base de tout reporting financier.

DailyPnL = NAV_t - NAV_(t-1)
CumPnL = NAV_t - NAV_0
📉

Volatilité 20j Annualisée

L'écart-type des rendements sur 20 jours, annualisé × √252. Mesure l'amplitude des fluctuations.

Vol = std(returns_20d) × √252
🌊

Max Drawdown

La pire chute depuis un sommet. Un drawdown de -15% signifie qu'on a perdu 15% depuis le pic historique.

DD = (NAV_t - Peak_t) / Peak_t
⚠️

VaR 95% (Value at Risk)

Le pire rendement journalier que tu subiras dans 95% des cas. Si VaR = -2%, tu perds moins de 2% 19 jours sur 20.

VaR_95 = percentile(returns, 5)
⭐

Sharpe Ratio

Le rendement par unité de risque. > 1 = excellent, > 0 = positif. Le standard de comparaison entre fonds.

Sharpe = Return / Volatility
(risk-free = 0)

👨‍💼
Senior · L'idempotence, c'est crucial !

Le pipeline analytics est idempotent : si tu le relances sur la même période, il purge et réinsère — jamais de doublons. C'est non-négociable en production. Les réglementations financières exigent que les chiffres soient reproductibles à l'identique. Un pipeline qui crée des doublons en cas de re-run est un bug critique, pas une feature manquante.

Intelligence Artificielle · Decision Support · LLM

L'Assistant IA — Décision Augmentée

Un copilote d'investissement qui analyse le contexte du portefeuille et génère des recommandations structurées — avec guardrails stricts pour la compliance.

👨‍💼
Senior · L'IA ne remplace pas le gérant, elle l'assiste !

Ce point est fondamental en finance. L'IA ici est un outil d'aide à la décision, pas un robot de trading autonome. MiFID II exige que toute décision soit tracée et justifiée. C'est pourquoi l'AI_AuditLog stocke chaque recommandation avec timestamp, score de confiance, et raisonnement. Les phrases comme "execute immediately" sont interdites par les guardrails — c'est une protection réglementaire.

🛡️ Les Guardrails (Compliance)

🚫

Phrases interdites : "execute immediately", "auto trade", "place order now". Si l'IA génère une de ces phrases, la recommandation est rejetée avant insertion SQL.

📏

Concentration max 40% : Aucune recommandation ne peut cibler un poids > 40% d'un titre dans le portefeuille. Règle de diversification réglementaire.

🎯

Confidence ≤ 0.95 : L'IA ne peut pas avoir plus de 95% de confiance — c'est un signal humain fort que la certitude absolue n'existe pas en finance.

🔟

Max 10 recommandations par briefing : Limite opérationnelle pour éviter le bruit et garder un signal clair pour le gérant.

Reporting · Power BI · Dashboards Finance

Power BI — La Couche de Présentation

Un modèle composite (Import + DirectQuery) connecté au Gold Layer SQL — des dashboards professionnels pour les gestionnaires et les équipes de contrôle.

👨‍💼
Senior · Composite Model — la bonne pratique BI

Un modèle composite Power BI = certaines tables en Import (chargées en mémoire — ultra-rapide pour les filtres) et d'autres en DirectQuery (requête SQL en temps réel — données fraîches). Les dimensions (DimDate, DimSecurity) vont en Import — elles changent peu. Les faits (FactTrades, AI tables) vont en DirectQuery — on veut voir les nouvelles données immédiatement. C'est l'architecture Power BI recommandée par Microsoft pour les modèles financiers.

CI/CD · Qualité · GitHub Actions · Tests

Pipeline CI/CD & Qualité

Chaque commit déclenche une validation automatique — tests Python, analyse SQL, et packaging de release. La qualité est non-négociable.

👨‍💼
Senior · Pourquoi le CI/CD est essentiel en finance ?

En finance, un bug en production peut coûter des millions. Le CI/CD est le filet de sécurité automatique — personne ne peut merger du code qui casse les tests. Dans un contexte réglementé (MiFID II, bâle III), tu dois prouver à l'auditeur que ton code a été validé, testé, et versionné. Le pipeline GitHub Actions est cette preuve. Sans CI/CD, tu n'as pas de projet data en production — tu as un prototype.

🐍

ci-python.yml

Déclenché à chaque push/PR sur les fichiers .py. Lance le compilateur Python et découvre les tests unitaires automatiquement.

on: push paths: ['**.py']
jobs:
✓ Python compile check
✓ unittest discover -s tests
✓ 4 suites de tests
🗄️

ci-sql.yml

Déclenché sur les fichiers .sql. Vérifie la syntaxe SQL et les conventions de nommage via le script validate_sql.py.

on: push paths: ['**.sql']
jobs:
✓ SQL syntax validation
✓ Naming conventions check
🚀

release.yml

Déclenché manuellement lors d'un tag de version. Génère le changelog, package la release et crée la note de version GitHub.

on: push tags: ['v*']
jobs:
✓ Version changelog
✓ Release packaging

👨‍💼
Senior · Le test du parsing décimal français, pourquoi c'est brillant

Boursorama exporte les prix en format français : "2 034,500" (espace milliers, virgule décimale). Si tu charges ça directement en float Python, tu obtiens une erreur ou — pire — une valeur fausse. Le test test_load_factprice_from_stg.py valide que le parser transforme correctement "2 034,500" → 2034.50. C'est le genre de bug silencieux qui coûte des millions en prod. Tester les cas limites, c'est l'ADN d'un bon data engineer.
